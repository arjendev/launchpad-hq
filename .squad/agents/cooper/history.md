# Project Context

- **Owner:** Arjen
- **Project:** launchpad — An npx package serving as a command and control center for managing multiple projects across repositories. Features a UI for project status overview, connected devcontainers with CLI sessions, kanban boards (todo, in progress, done), deep introspection of copilot sessions, and ability to attach to ongoing sessions. Potentially leveraging the GitHub Copilot SDK.
- **Stack:** Node.js, TypeScript, npx CLI, Terminal/Web UI, GitHub API, Copilot SDK, Devcontainer API
- **Created:** 2026-03-13

- **2026-03-14:** Full SDK interface audit completed (`.squad/agents/cooper/sdk-audit.md`). The `@github/copilot-sdk` (protocol v3) exposes a massive surface: CopilotClient has 19 public methods, CopilotSession has 16 methods/properties, SessionConfig has 20+ fields, there are 55+ distinct session event types, server-scoped RPC has 4 namespaces (ping/models/tools/account), and session-scoped RPC has 8 namespaces (model/mode/plan/workspace/fleet/agent/compaction/tools/permissions). We currently expose roughly 15% of this. The adapter layer (adapter.ts + sdk-adapter.ts) is a lossy abstraction — it re-declares SDK types, uses `any` everywhere, renames events unnecessarily, and flattens the rich typed event union into `Record<string, unknown>`. Recommendation: delete the adapter, import SDK types directly in the manager, forward events unchanged over the wire, and let TypeScript enforce correctness. Key missing capabilities by priority: model listing, auth status, mode control, non-blocking send (vs sendAndWait), permission UI flow, user input handler, assistant.intent events, usage/quota tracking, session.shutdown stats, plan RPC, and client lifecycle events.

- **2025-07-22:** Completed final security audit of the full codebase. Key findings: (1) No critical vulnerabilities — token generation, constant-time validation, and challenge-nonce daemon auth are all solid. (2) Two HIGH issues for tunnel scenarios: REST API has zero authentication (anyone with tunnel URL has full control), and `GET /api/settings` leaks the WS session token in its response. (3) No auth handshake timeout on pending daemon connections (DoS vector). (4) Daemon `--token` CLI arg is visible in `ps` output. (5) Preview proxy has no port validation on daemon-reported previewPort (SSRF risk). (6) No rate limiting, no CSP headers, no WS origin validation. (7) Clean dependency tree (0 npm audit vulnerabilities). (8) Input validation is consistent throughout — owner/repo regex, path traversal blocks, port range checks. Priority fix order: REST API auth → session token leak → handshake timeout → WS origin validation. Full report at `.squad/decisions/inbox/cooper-security-review-opus.md`.

- **2026-03-15:** Final security review found real release blockers in the current control plane. The biggest pattern: the code uses good crypto primitives (`randomBytes`, `timingSafeEqual`, nonce challenge) but then leaves the surrounding HTTP surface effectively unauthenticated, including endpoints that leak `sessionToken` and `daemonToken`. Devcontainer `0.0.0.0` binding and Dev Tunnel publishing turn a local-trust design into a remotely reachable one, so control-plane auth has to come before any public release. Also notable: daemon Copilot sessions currently run with `onPermissionRequest: approveAll`, meaning prompt/control-plane compromise can escalate straight into silent tool execution.

### Session — Deep Research: Devcontainer Lifecycle Management (requested by Arjen)

- **Task:** Research spike on how launchpad-hq can fully manage the underlying devcontainer lifecycle.
- **Scope:** 6 research areas — devcontainer CLI/APIs, non-devcontainer codebases, VS Code server options, cross-platform Docker, cloud container services, architecture options.
- **Key findings:**
  - `devcontainer` CLI provides full lifecycle (up/down/exec/stop) with JSON output — ideal for programmatic use.
  - `dockerode` (Node.js Docker SDK) provides low-level health monitoring, events, stats, and remote Docker host support.
  - DevPod (loft.sh) is the most strategically relevant tool — client-only, provider-agnostic, uses devcontainer.json, open source.
  - openvscode-server is the best option for embedding VS Code in HQ's web UI (MIT, Docker-ready, tracks VS Code releases).
  - Docker-only is pragmatic for Phase 1; Podman/nerdctl have gaps in devcontainer support.
  - fly.io is the most cost-effective cloud backend for dev environments (stop-to-zero, fast resume).
- **Recommendation:** Phased approach — Phase 1: Local Docker via devcontainer CLI + dockerode. Phase 2: Remote Docker + openvscode-server. Phase 3: DevPod integration or custom provider abstraction.
- **6 architecture options presented:** (A) Local Docker Only, (B) +Remote Docker, (C) DevPod Integration, (D) K8s-based, (E) Hybrid local+cloud, (F) Custom abstraction.
- **No code changes in research phase.** Issue #74 filed with full research findings and phased roadmap.
