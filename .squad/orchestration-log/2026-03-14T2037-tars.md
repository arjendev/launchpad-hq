### Deliverables

1. **Authentication Models Documentation**: Detailed analysis of 4 primary access control models:
   - **Authenticated (Default)**: User logs in with Microsoft/GitHub (high security, requires login)
   - **Anonymous**: Anyone with tunnel URL can access (low security, simplest)
   - **Token-Based**: Short-lived bearer tokens with optional expiration (recommended for QR)
   - **Organization-Level**: Grant access to GitHub org or Microsoft tenant members (team-based)

2. **DevTunnel API Surface Findings**:
   - No official Node.js SDK available
   - No public REST API published
   - CLI is canonical interface (stable output format)
   - Child process + stdout parsing is reliable integration pattern

3. **QR Code Implementation Roadmap**:
   - **Phase 1 (P2)**: Pre-login model with temporary tunnel (already decided by Cooper)
   - **Phase 2 (P3)**: Add `devtunnel token` generation for passwordless mobile access
   - **Phase 3 (P4+)**: Org/tenant access control, persistent tunnels

4. **Implementation Architecture for Token-Based Auth**:
   ```
   Desktop: devtunnel host → devtunnel token → QR(url + token)
   Phone:   scan QR → request with token → devtunnel relay validates → launchpad session
   ```

## Key Findings

### Phase 1 (P2) - Current Grooming
✅ Pre-login model is correct for MVP:
- User runs `devtunnel user login` once
- Launchpad assumes user is authenticated
- Return `status: "auth_failed"` with guidance if token missing
- **Security:** High (system keychain, no token storage)

### Phase 2 (P3) - QR Code Enhancement
✅ Token-based auth is **best path**:
- Call `devtunnel token TUNNELID --expiration 4h` after tunnel creation
- Embed token in QR: `https://tunnel.devtunnels.ms:3000?access_token=TOKEN`
- Phone scans → instant access without login
- Token auto-expires; devtunnel relay validates automatically
- Multiple concurrent tokens supported

### Alternatives Evaluated
❌ **Anonymous (`--allow-anonymous`)**: Viable for P2 MVP only if tunnel auto-deletes and expires; publicly accessible (medium security)

❌ **Pre-login for QR**: Defeats purpose (scan → requires phone login → defeats UX benefit)

## Architecture Decisions

| Layer | Choice | Rationale |
|-------|--------|-----------|
| **SDK** | CLI wrapper (child_process) | No official SDK; CLI is stable and canonical |
| **Integration** | Adapt self-daemon spawner pattern | Proven: stdio piping, exit handling, graceful shutdown |
| **Phase 1 Auth** | Pre-login (user responsibility) | Simple; aligns with Cooper's grooming |
| **Phase 2+ Auth** | Token-based with short expiration | Secure, instant mobile access, no backend state needed |
| **Token Model** | Stateless, validates at devtunnel relay | Leverage existing infrastructure; no token storage |

## Design Principles Extracted

1. **Leverage devtunnel's security model** — Don't reinvent auth; use relay as security boundary
2. **Token expiration is automatic** — Short-lived tokens (4h–30d) reduce risk
3. **QR code is trust anchor** — Tokens embedded in QR are only accessible to intended recipients
4. **Session mapping is simple** — Launchpad validates token presence, creates local session
5. **Multi-port support is native** — Devtunnel CLI supports multiple ports; future enhancement
