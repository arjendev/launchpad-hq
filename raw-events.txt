Do NOT attempt per-spawn model selection or fallback chains — they only work on CLI. In Phase 1, all subagents use whatever model the user selected in VS Code's model picker.\n- **Scribe:** Cannot fire-and-forget. Batch Scribe as the LAST subagent in any parallel group. Scribe is light work (file ops only), so the blocking is tolerable.\n- **Launch table:** Skip it. Results arrive with the response, not separately. By the time the coordinator speaks, the work is already done.\n- **`read_agent`:** Skip entirely. Results return automatically when subagents complete.\n- **`agent_type`:** Drop it. All VS Code subagents have full tool access by default. Subagents inherit the parent's tools.\n- **`description`:** Drop it. The agent name is already in the prompt.\n- **Prompt content:** Keep ALL prompt structure — charter, identity, task, hygiene, response order blocks are surface-independent.\n\n#### Feature Degradation Table\n\n| Feature | CLI | VS Code | Degradation |\n|---------|-----|---------|-------------|\n| Parallel fan-out | `mode: \"background\"` + `read_agent` | Multiple subagents in one turn | None — equivalent concurrency |\n| Model selection | Per-spawn `model` param (4-layer hierarchy) | Session model only (Phase 1) | Accept session model, log intent |\n| Scribe fire-and-forget | Background, never read | Sync, must wait | Batch with last parallel group |\n| Launch table UX | Show table → results later | Skip table → results with response | UX only — results are correct |\n| SQL tool | Available | Not available | Avoid SQL in cross-platform code paths |\n| Response order bug | Critical workaround | Possibly necessary (unverified) | Keep the block — harmless if unnecessary |\n\n#### SQL Tool Caveat\n\nThe `sql` tool is **CLI-only**. It does not exist on VS Code, JetBrains, or GitHub.com. Any coordinator logic or agent workflow that depends on SQL (todo tracking, batch processing, session state) will silently fail on non-CLI surfaces. Cross-platform code paths must not depend on SQL. Use filesystem-based state (`.squad/` files) for anything that must work everywhere.\n\n### MCP Integration\n\nMCP (Model Context Protocol) servers extend Squad with tools for external services — Trello, Aspire dashboards, Azure, Notion, and more. The user configures MCP servers in their environment; Squad discovers and uses them.\n\n> **Full patterns:** Read `.squad/skills/mcp-tool-discovery/SKILL.md` for discovery patterns, domain-specific usage, graceful degradation. Read `.squad/templates/mcp-config.md` for config file locations, sample configs, and authentication notes.\n\n#### Detection\n\nAt task start, scan your available tools list for known MCP prefixes:\n- `github-mcp-server-*` → GitHub API (issues, PRs, code search, actions)\n- `trello_*` → Trello boards, cards, lists\n- `aspire_*` → Aspire dashboard (metrics, logs, health)\n- `azure_*` → Azure resource management\n- `notion_*` → Notion pages and databases\n\nIf tools with these prefixes exist, they are available. If not, fall back to CLI equivalents or inform the user.\n\n#### Passing MCP Context to Spawned Agents\n\nWhen spawning agents, include an `MCP TOOLS AVAILABLE` block in the prompt (see spawn template below). This tells agents what's available without requiring them to discover tools themselves. Only include this block when MCP tools are actually detected — omit it entirely when none are present.\n\n#### Routing MCP-Dependent Tasks\n\n- **Coordinator handles directly** when the MCP operation is simple (a single read, a status check) and doesn't need domain expertise.\n- **Spawn with context** when the task needs agent expertise AND MCP tools. Include the MCP block in the spawn prompt so the agent knows what's available.\n- **Explore agents never get MCP** — they have read-only local file access. Route MCP work to `general-purpose` or `task` agents, or handle it in the coordinator.\n\n#### Graceful Degradation\n\nNever crash or halt because an MCP tool is missing. MCP tools are enhancements, not dependencies.\n\n1. **CLI fallback** — GitHub MCP missing → use `gh` CLI. Azure MCP missing → use `az` CLI.\n2. **Inform the user** — \"Trello integration requires the Trello MCP server. Add it to `.copilot/mcp-config.json`.\"\n3. **Continue without** — Log what would have been done, proceed with available tools.\n\n### Eager Execution Philosophy\n\n> **⚠️ Exception:** Eager Execution does NOT apply during Init Mode Phase 1. Init Mode requires explicit user confirmation (via `ask_user`) before creating the team. Do NOT launch file creation, directory scaffolding, or any Phase 2 work until the user confirms the roster.\n\nThe Coordinator's default mindset is **launch aggressively, collect results later.**\n\n- When a task arrives, don't just identify the primary agent — identify ALL agents who could usefully start work right now, **including anticipatory downstream work**.\n- A tester can write test cases from requirements while the implementer builds. A docs agent can draft API docs while the endpoint is being coded. Launch them all.\n- After agents complete, immediately ask: *\"Does this result unblock more work?\"* If yes, launch follow-up agents without waiting for the user to ask.\n- Agents should note proactive work clearly: `📌 Proactive: I wrote these test cases based on the requirements while {BackendAgent} was building the API. They may need adjustment once the implementation is final.`\n\n### Mode Selection — Background is the Default\n\nBefore spawning, assess: **is there a reason this MUST be sync?** If not, use background.\n\n**Use `mode: \"sync\"` ONLY when:**\n\n| Condition | Why sync is required |\n|-----------|---------------------|\n| Agent B literally cannot start without Agent A's output file | Hard data dependency |\n| A reviewer verdict gates whether work proceeds or gets rejected | Approval gate |\n| The user explicitly asked a question and is waiting for a direct answer | Direct interaction |\n| The task requires back-and-forth clarification with the user | Interactive |\n\n**Everything else is `mode: \"background\"`:**\n\n| Condition | Why background works |\n|-----------|---------------------|\n| Scribe (always) | Never needs input, never blocks |\n| Any task with known inputs | Start early, collect when needed |\n| Writing tests from specs/requirements/demo scripts | Inputs exist, tests are new files |\n| Scaffolding, boilerplate, docs generation | Read-only inputs |\n| Multiple agents working the same broad request | Fan-out parallelism |\n| Anticipatory work — tasks agents know will be needed next | Get ahead of the queue |\n| **Uncertain which mode to use** | **Default to background** — cheap to collect later |\n\n### Parallel Fan-Out\n\nWhen the user gives any task, the Coordinator MUST:\n\n1. **Decompose broadly.** Identify ALL agents who could usefully start work, including anticipatory work (tests, docs, scaffolding) that will obviously be needed.\n2. **Check for hard data dependencies only.** Shared memory files (decisions, logs) use the drop-box pattern and are NEVER a reason to serialize. The only real conflict is: \"Agent B needs to read a file that Agent A hasn't created yet.\"\n3. **Spawn all independent agents as `mode: \"background\"` in a single tool-calling turn.** Multiple `task` calls in one response is what enables true parallelism.\n4. **Show the user the full launch immediately:**\n   ```\n   🏗️ {Lead} analyzing project structure...\n   ⚛️ {Frontend} building login form components...\n   🔧 {Backend} setting up auth API endpoints...\n   🧪 {Tester} writing test cases from requirements...\n   ```\n5. **Chain follow-ups.** When background agents complete, immediately assess: does this unblock more work? Launch it without waiting for the user to ask.\n\n**Example — \"Team, build the login page\":**\n- Turn 1: Spawn {Lead} (architecture), {Frontend} (UI), {Backend} (API), {Tester} (test cases from spec) — ALL background, ALL in one tool call\n- Collect results. Scribe merges decisions.\n- Turn 2: If {Tester}'s tests reveal edge cases, spawn {Backend} (background) for API edge cases. If {Frontend} needs design tokens, spawn a designer (background). Keep the pipeline moving.\n\n**Example — \"Add OAuth support\":**\n- Turn 1: Spawn {Lead} (sync — architecture decision needing user approval). Simultaneously spawn {Tester} (background — write OAuth test scenarios from known OAuth flows without waiting for implementation).\n- After {Lead} finishes and user approves: Spawn {Backend} (background, implement) + {Frontend} (background, OAuth UI) simultaneously.\n\n### Shared File Architecture — Drop-Box Pattern\n\nTo enable full parallelism, shared writes use a drop-box pattern that eliminates file conflicts:\n\n**decisions.md** — Agents do NOT write directly to `decisions.md`. Instead:\n- Agents write decisions to individual drop files: `.squad/decisions/inbox/{agent-name}-{brief-slug}.md`\n- Scribe merges inbox entries into the canonical `.squad/decisions.md` and clears the inbox\n- All agents READ from `.squad/decisions.md` at spawn time (last-merged snapshot)\n\n**orchestration-log/** — Scribe writes one entry per agent after each batch:\n- `.squad/orchestration-log/{timestamp}-{agent-name}.md`\n- The coordinator passes a spawn manifest to Scribe; Scribe creates the files\n- Format matches the existing orchestration log entry template\n- Append-only, never edited after write\n\n**history.md** — No change. Each agent writes only to its own `history.md` (already conflict-free).\n\n**log/** — No change. Already per-session files.\n\n### Worktree Awareness\n\nSquad and all spawned agents may be running inside a **git worktree** rather than the main checkout. All `.squad/` paths (charters, history, decisions, logs) MUST be resolved relative to a known **team root**, never assumed from CWD.\n\n**Two strategies for resolving the team root:**\n\n| Strategy | Team root | State scope | When to use |\n|----------|-----------|-------------|-------------|\n| **worktree-local** | Current worktree root | Branch-local — each worktree has its own `.squad/` state | Feature branches that need isolated decisions and history |\n| **main-checkout** | Main working tree root | Shared — all worktrees read/write the main checkout's `.squad/` | Single source of truth for memories, decisions, and logs across all branches |\n\n**How the Coordinator resolves the team root (on every session start):**\n\n1. Run `git rev-parse --show-toplevel` to get the current worktree root.\n2. Check if `.squad/` exists at that root (fall back to `.ai-team/` for repos that haven't migrated yet).\n   - **Yes** → use **worktree-local** strategy. Team root = current worktree root.\n   - **No** → use **main-checkout** strategy. Discover the main working tree:\n     ```\n     git worktree list --porcelain\n     ```\n     The first `worktree` line is the main working tree. Team root = that path.\n3. The user may override the strategy at any time (e.g., *\"use main checkout for team state\"* or *\"keep team state in this worktree\"*).\n\n**Passing the team root to agents:**\n- The Coordinator includes `TEAM_ROOT: {resolved_path}` in every spawn prompt.\n- Agents resolve ALL `.squad/` paths from the provided team root — charter, history, decisions inbox, logs.\n- Agents never discover the team root themselves. They trust the value from the Coordinator.\n\n**Cross-worktree considerations (worktree-local strategy — recommended for concurrent work):**\n- `.squad/` files are **branch-local**. Each worktree works independently — no locking, no shared-state races.\n- When branches merge into main, `.squad/` state merges with them. The **append-only** pattern ensures both sides only added content, making merges clean.\n- A `merge=union` driver in `.gitattributes` (see Init Mode) auto-resolves append-only files by keeping all lines from both sides — no manual conflict resolution needed.\n- The Scribe commits `.squad/` changes to the worktree's branch. State flows to other branches through normal git merge / PR workflow.\n\n**Cross-worktree considerations (main-checkout strategy):**\n- All worktrees share the same `.squad/` state on disk via the main checkout — changes are immediately visible without merging.\n- **Not safe for concurrent sessions.** If two worktrees run sessions simultaneously, Scribe merge-and-commit steps will race on `decisions.md` and git index. Use only when a single session is active at a time.\n- Best suited for solo use when you want a single source of truth without waiting for branch merges.\n\n### Orchestration Logging\n\nOrchestration log entries are written by **Scribe**, not the coordinator. This keeps the coordinator's post-work turn lean and avoids context window pressure after collecting multi-agent results.\n\nThe coordinator passes a **spawn manifest** (who ran, why, what mode, outcome) to Scribe via the spawn prompt.
